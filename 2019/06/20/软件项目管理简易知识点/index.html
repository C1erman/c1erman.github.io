软件项目管理概念

项目

项目是为完成某一独特的产品或服务所做的一次性的、有限的努力。

项目的特征

  • 目标性
  • 相关性
  • 周期性
  • 独特性
  • 约束性

每一个项目的资源、成本、时间都是一定的。

  • 不确定性

项目管理的五个标准过程组

  1. 启动过程组
  2. 计划过程组
  3. 执行过程组
  4. 控制过程组
  5. 收尾过程组

软件项目与一般项目的区别

软件项目除了具备一般项目的基本特征之外，还具有以下特点：

  1. 软件是一种逻辑实体，具有抽象性。
  2. 软件开发的成本相当昂贵。
  3. 软件没有机械硬件的磨损和老化问题。

软件项目管理

为了使软件项目按照预定的成本、进度、质量顺利完成，而对成本、人员、进度进行分析和管理的活动。

软件项目的解决方案

  1. 自主开发
  2. 完全外包
  3. 购买商用软件+自主开发

软件项目范围

  1. 产品范围

产品或服务所包含的特征或功能。

  1. 项目范围

为交付具有规定特征和功能的产品所必须完成的工作。

可行性分析

  1. 技术可行性
  2. 经济可行性
  3. 社会可行性
  4. 操作可行性

合同类型

  1. 成本补偿合同
  2. 功能计费点合同
  3. 固定单价合同
  4. 固定总价合同

上述合同从上到下，对软件开发方的风险依此增大。

合同计划

软件项目需求管理

软件需求

用户对软件的功能和性能的要求，是一个系统或程序开发工作的说明，表现形式一般为文档形式。

软件需求包括功能性需求和非功能性需求。

功能性需求又可以分为三个层次：

  1. 业务需求
  2. 用户需求
  3. 功能需求

需求开发

  1. 需求获取
  2. 需求分析
  3. 需求规格编写
  4. 需求验证

这四个阶段不一定遵循线性顺序，它们是互相独立和反复的。

软件项目进度管理

由进度计划与进度控制组成。

WBS：制定进度计划的基础

将一个复杂的开发系统分层逐步细化为一个个工作任务单元。

  • 产品导向型

以可交付产品作为划分依据。

  • 活动导向型

以项目活动作为划分依据。

甘特图

优点：清楚的表明了项目的计划进度，能动态的反映当前开发紧张状况。

缺点：不能表达出各任务之间复杂的逻辑关系。

网络图

展示项目之间的各个活动以及活动之间的逻辑关系。

常用的网络图有PDM、ADM。

每个节点必须有唯一的编号。

网络图：PDM

节点法。

  • 节点表示活动。
  • 箭头表示工作的先后顺序和相互关系。

有四种逻辑关系：

  • 结束开始
  • 结束结束
  • 开始结束
  • 开始开始

网络图：ADM

箭线法。

  • 节点表示事件。
  • 箭头表示活动，活动的描述写在箭头上。

紧前事件：箭头之前的事件。

紧后事件：箭头之后的事件。

只有一种逻辑关系：结束开始。

使用虚活动来表明活动之间的逻辑关系，它既不消耗资源也不占用时间，用虚线箭头表示。

甘特图与网络图差异

如下表所述：

网络图 甘特图
活动之间的逻辑关系 可以反应 只能反应简单的逻辑关系
项目的完整工作流程 只有一个起点、一个终点，可以反应 起点和终点不集中在一个点上，不容易从中看出完整的工作流程
计划与组织项目工作 能起较大作用 作用不大

计划评审

加权得到的平均期望值为：(最乐观期望值O+4*最可能期望值M+最悲观期望值P)/6。

标准差为：(P-O)/6。

关键路径法

项目中总工期最长的那条线。

  • 最早开始时间：项目的第一个活动的最早开始时间为0，其余为紧前活动的最早结束时间。
  • 最早结束时间：本身的最早开始时间+活动历时。
  • 最晚结束时间：项目的最后一个活动的最晚结束时间为最早结束时间，其余为紧后活动的最晚开始时间。

当有多个后置任务时，选取其中最小的最晚开始时间。

  • 最晚开始时间：本身的最晚结束时间-活动历时。

项目中最早开始时间 == 最晚开始时间的活动，为关键活动，组成的路径即为关键路径。

项目估算

成本估算方法

  • 基于分解技术的方法

功能点估算法、代码行估算法。

  • 基于经验模型的方法

COCOMO模型、IBM模型。

工期估算方法

  • 类比估算法

将历史项目的实际成本作为当前项目的基础，根据当前项目的范围、规模来估算当前项目的成本。是自顶向下估算方法的一种类型。

当项目信息足够时，使用类比估算法。

  • 德尔菲法

由多位专家对项目进行估算，专家对项目理解的程度是该方法的关键点。

当获得的项目信息有限时，使用德尔菲法。

质量管理

软件质量模型

  • Boehm模型

可移植性、可使用性、可维护性。

  • McCall模型

产品运行、产品修正、产品转移。

  • ISO 9126

内部质量模型、外部质量模型、使用中质量模型。

质量管理步骤

  • 质量计划

是决定如何满足质量标准的计划安排和方法。

  • 质量保证

证明项目将会达到有关质量标准的工程活动，包括项目产品审计和项目执行过程审计。

  • 质量控制

对阶段性的结果进行验证，为质量保证提供依据。

质量管理评审

即是过程评审也是产品评审，对产品组件或文档进行技术分析，旨在发现设计或者代码中的详细错误。

是质量管理过程中的最后一个阶段。

质量管理评审方法

同行评审、走查、会议审查。

风险管理

风险管理

  • 风险识别
  • 风险评估
  • 风险处理
  • 风险监控

风险识别的依据

风险管理规划、项目规划、历史资料、风险种类、制约因素与假定条件。

风险管理的意义和作用

作用：

  • 促进决策实施的科学化、合理化，降低决策的风险水平
  • 保障项目经营目标的顺利实现

意义：

  • 有利于提高全社会的资金利用效率，促进产业结构的优化
  • 有利于社会主义市场经济的健康发展

人力资源和沟通管理

项目组织结构

  • 职能型

是应用最广泛的组织结构。

具有明确的等级划分。

优点：资源利用灵活，成本较低，有利于整体协调。

缺点：项目经理权力弱化，项目组成员责任淡化。

适用于公司内部项目。

  • 项目型

项目成员有项目完成后无家可归的担忧。

项目的组织独立于公司的职能部门之外，以项目经理为首。

优点：项目经理权力集中，有利于资源调动，结构简单，沟通效率高。

缺点：资源无法共享，项目间缺少交流，阻碍技术水平提升。

适用于大型项目。

  • 矩阵型

是上述两者的混合体，每个职员有两个经理——项目经理和职能经理。

项目经理负责整合公司资源完成项目目标。

职能经理对高质量的完成所承担的产品任务负责。

可分为以下三种。

弱矩阵型组织结构：保持了较多职能型组织结构的特征，适用于技术简单的项目。

平衡型矩阵组织结构：任命参与项目的一位职员为项目经理，有一定权力，适用于中等技术复杂程度的项目。

强矩阵型组织结构：具有较大权限的项目经理以及专职的项目管理人员，适用于技术复杂的项目。

项目经理的职权

  • 独立决策权
  • 项目团队的组建权
  • 项目实施的控制权
  • 项目资金的控制权

项目收尾

包括合同收尾和管理收尾。

  • 合同收尾

与客户比照合同，审核是否完成了所要求的内容。结束合同并理清账目。

  • 管理收尾

对于项目组内部，把做好的项目文档、代码、与客户交流的文件等归档保存。

项目收尾的重要性

  • 是项目管理的重要评审点

  • 是沟通管理的契机

  • 是项目收尾记录的基础

项目验收

对软件项目成果的检验和确认，是一个循序渐进的过程：准备验收材料、提交申请、初审、复审、验收合格、完成移交工作。

包括：

  • 软件系统验收
  • 质量验收

对软件系统的功能、性能、用户操作界面、可维护性等方面进行验收。

  • 资料验收

对开发文档、管理文档、用户文档进行验收，检查项目资料的完整性、书写是否规范、内容是否一致。

项目评价的内容

  • 项目计划

根据项目目标的要求，对项目范围内的各项活动做出合理的安排。

  • 项目组织

为了进行项目管理，完成预定的项目工作，而建立的管理结构。

  • 项目效益

项目实施完成后对项目预期的近期经济效益和远期影响的评价。

  • 成本控制

将此项目实际发生的费用与项目批准的实施费用预算进行对比和评价。

  • 进度控制

以项目实际的进度与计划的进度进行对比和评价。
