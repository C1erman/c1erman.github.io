题外话：URL与URI

与URIの统一资源标识符Uniform Resourse Identifier相比，我们显然更加熟悉URLの统一资源定位符Uniform Resource Locator。

对URI来说：

  • Uniform：规定统一的协议方案，以处理不同类型的资源
  • Resource：可标识的任何东西，可以是单一的，也可以是多数的集合体
  • Identifier：表示可标识的对象

综上所述，URI就是由某个协议方案表示的资源的定位标识符。

采用HTTP协议时，协议方案就是http。除此之外，还有ftp、telnet等。

什么是HTTP

HTTP是超文本传输协议Hyper Text Transfer Protocal的简称。它从属于计算机网络体系结构中的最高层——应用层。

应用层协议定义的是应用进程间通信和交互的规则。

它只是计算机网络中的很小一部分。

为了能够详细的介绍各层的内容，有人抽象出了五层的结构：

这五层的作用如下所述：

  • 应用层：是体系结构中的最高层，它的任务是通过应用进程间的交互来完成特定网络应用。称其数据单元为报文
  • 运输层：任务为负责向两台主机中的进程之间的通信提供通用的数据传输服务。主要使用面向连接、可靠的TCP协议和无连接、尽最大努力交付的UDP协议
  • 网络层：负责为分组交换网上的不同主机提供通信服务，因为采用了IP协议包装了报文段——IP数据报，也可以叫做IP层
  • 数据链路层：将上层得到的IP数据报封装成帧，采用可靠的协议，通过不同的信道进行两台主机之间的数据传输，同时也提供了帧的差错检验机制防止帧与帧之间的碰撞
  • 物理层：以比特为传输单位，采用某种复用方式，如码分复用进行单工或双工通信

HTTP报文

用于HTTP协议交互的信息就是HTTP报文。

HTTP报文由以下部分组成：

  • 报文首部：服务器端或客户端需处理的请求或响应的内容及属性
  • 空行CR+LF：CR为Carriage Return回车符，LF为Line Feed换行符
  • 报文主体：应该被发送的数据

维持连接

完成一次HTTP请求/响应后，客户端/服务器并不是马上断开与对方的连接，在HTTP1.1中，Connection: keep-alive是默认启用的，表示持久连接以提高网络的吞吐能力。

注意，HTTPのkeep-alive和 TCPのkeep-alive虽然都是一种保活机制，但是它们完全不相同，一个作用于应用层，一个作用于传输层。

解

大致了解计算机网络的各层之后，我们来看看在浏览器中输入URL后到底发生了什么。

1.DNS解析

互联网的每一台主机都被分配了IP地址，可它并不方便记忆，而方便记忆的是网址，从域名到IP地址的转换就是DNS解析。

浏览器向本地DNS服务器发出请求，对一个新域名，本地DNS服务器没有缓存，不能直接将域名转为IP地址，需要采用递归、迭代的方式，依此向根域名服务器.、顶级域名服务器.com、权威域名服务器发起查询请求。

DNS优化

DNS存在着多级缓存，大致有这么几种：浏览器缓存、系统缓存、路由器缓存等。

  • Firefox浏览器可以使用about:config查看相关配置，但它参考系统缓存
  • Linux系统的缓存存在于/etc/host中

DNS负载均衡

CDN服务，即Content Delivery Network，就是利用DNS的重定向技术，返回一个和用户链接上最接近的IP地址给用户。

2.TCP连接：大好きなTCP三报文握手と四报文挥手

在知道服务器IP地址之后，客户端——即浏览器，将通过构建HTTP请求报文并通过TCP协议发送到服务器指定端口进行通信：HTTP协议为80/8080端口，HTTPS协议为443端口。

这可谓是最最常遇到的了。

3.发送HTTP请求

建立好连接后，浏览器开始发送HTTP请求。需要注意的是，HTTP请求头是不区分大小写的，它基于ASCII进行编码。

常见请求方法

截止到HTTP1.1常用的请求方法如下：

  • GET：GET请求会显式指定请求的资源，此时HTTP请求报文将不含请求体
  • POST：向指定资源提交数据进行处理请求，数据被包含在请求体中。POST可能会导致新的资源建立或修改已有资源
  • PUT：向指定资源上传最新内容，以取代服务器端指定的内容，它是幂等的，即：任意多次执行所产生的影响均与一次执行的影响相同
  • DELETE：请求服务器删除所请求的URI标识的资源，它也是幂等的
  • HEAD：类似于GET请求，不过返回的响应无具体内容，只用于获取响应报头

4.返回HTTP响应

服务器接收并处理完请求，返回HTTP响应。
