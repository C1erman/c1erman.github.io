跨域

发表于 2019-04-28 | 更新于 2019-08-05 | 分类于 -前端知识

跨域问题，基本上每位前端开发人员都会遇到。

本篇文章将讲述什么是跨域，顺带提一提JSON与JSONP这两个并不相近的东西。

跨域

同源策略

同源策略是浏览器安全性的基本保证，当协议、域名、端口三者都相同时才视为同源，需要注意的是，即使不同的域名指向相同的ip地址，也并非同源。

最常见的非同源问题就是http://localhost:8080与http://172.0.0.1:8080，虽然都指向本机。

localhost与172.0.0.1的区别：

localhost不通过网卡传输，不受网卡和防火墙限制，访问时带着本机当前用户的权限。

172.0.0.1则依赖于网卡，会受到网卡和防火墙限制，由于是通过网络进行访问，涉及到网络用户权限的问题。

使用localhost访问时不会解析成ip地址，因此不会占用网络资源。

同源策略限制内容

  • Cookie、localStorage、IndexedDB等存储性数据
  • DOM节点
  • 跨域Ajax请求

Ajax：即Asynchronous Javascript And XML，异步JavaScript与XML。

解决方案

  • JSONP
  • CORS
  • postMessage
  • websocket

JSON

JSON，即JavaScript Object Notation，是纯文本结构的数据存储。它是一种数据交换格式，被广泛应用于WEB信息交换中。

JSON将数据存储为纯文本格式，它的语法是JavaScript表达式语法的一个子集。

如果你熟悉JavaScript的话，那么你就能立刻掌握JSON的格式，唯一需要注意的是：JSON字符串必须使用双引号。

关于JSON的全部语法，可以到这个站点学习。

JSON之所以流行，一个比较重要的原因是，可以把JSON数据结构解析为有用的JavaScript对象。

JSON对象有两个方法：stringify()和parse()：

  • JSON.stringify(value[, replacer[, space]])方法是将一个JavaScript值转换为一个JSON字符串。后两个参数都是可选的，如果指定replacer是一个数组，则结果中只包含数组中列出来的属性；如果是函数，则该函数拥有属性名、属性值这两个参数用于做替换操作。而space则用于控制结果中的缩进和空白符，最大值为10。

需要注意的是，在序列化JavaScript对象时，值为undefined的任何属性、非自有属性都会被跳过。

  • JSON.parse(text[, reviver])方法用于将JSON字符串构造为字符串描述的JavaScript对象。可选参数reviver为一个函数，该函数的两个参数：属性名、属性值用来修改解析生成的原始值。

如果reviver函数返回undefined，则表明删除某一属性。

JSONP

JSONP，即JSON With Paddding，年纪已经很大了，它是早先流行的跨域数据交互方法。

在基于Ajax的通信机制中，一个主要限制就是跨域安全策略。默认情况下，XMLHttpRequest对象只能访问与包含它的页面位于同一个域中的资源。

这种策略显然是一种安全策略，可以预防某些恶意行为。

但是，实现合理的跨域请求也是很有必要的。

有人发现：HTML页面上，凡是拥有src属性的标签，像<script />、<img />等，都拥有跨域的能力。

于是，客户端通过调用JavaScript脚本的方式，来调用服务器动态生成JSON文件，再交由客户端的特定函数处理以便获取数据，这样，通信便完成了。

通过动态添加<script />标签，并设置其src地址，来调用服务器指定的回调函数以得到数据传输，这就是JSONP。

CORS

JSONP也有不足之处，诸如：只能使用GET方法、可能会在返回中夹带恶意代码等。

另一种跨域方法就是使用CORS，所有的浏览器都支持这个W3C规范，欲了解详情，请参考MDN上的HTTP访问控制。

极其推荐每位WEB开发者阅读上述文章。

CORS，跨域资源共享，即Cross Origin Resource Sharing 。

它允许某一页面向跨域服务器发送并接受请求。

简单请求

某些请求并不会触发CORS预检请求，我们称这种请求为简单请求。

注意：简单请求并不是由W3C规范所定义的。

若某个请求满足以下两个条件，则被称作简单请求。

  • 使用GET、POST、HEAD方法之一
  • 不人为设置CORS安全首部字段集合之外的其它首部字段

CORS安全首部字段集合：

Accept：向服务器告知客户端能够处理的MIME类型

Accept-Language：允许客户端声明其能够理解的自然语言及有限选择的区域语言

Content-Language：为实体首部，用于指定页面的目标受众

Content-Type：实体首部，指示实际发送的数据类型，在此被限制为以下三种：

  • text/plain
  • multipart/form-data
  • application/x-www-form-urlencoded

DPR：尚处于草案阶段

Downlink：尚处于草案阶段

Save-Data：尚处于草案阶段

Viewport-Width：尚处于草案阶段

Width：尚处于草案阶段

简单请求的基本流程

对于简单请求，浏览器会直接发出CORS请求，即在请求头中增加Origin字段，表明请求来源的站点。

该字段仅指示服务器名称，并不包含任何路径信息。

请求发送至服务器后，返回响应至浏览器，这时，如果浏览器发现响应头中，不包含Access-Control-Allow-Origin字段，则会拦截该响应，同时抛出错误。

即：跨站请求正常发起了，但是响应结果被浏览器拦截了。

如果Origin字段指定的源在服务器的许可范围内，服务器返回的响应头将会至多添加下述三个关联字段：

  • Access-Control-Allow-Origin

该字段指定了响应的资源是否被允许与指定的Origin共享。

可取值为：所有*、给定域名http:seefun.club。

  • Access-Control-Allow-Credentials

该字段指定了是否可以将对请求的响应暴露给页面。

可取值为：是true。

比起将该字段的值设置为false，忽略对该字段的赋值更好。

  • Access-Control-Expose-Headers

该字段列出了哪些头部字段可以作为响应的一部分暴漏给外部。

默认情况下，只有六种简单响应头部，即simple response headers，可以暴漏给外部。

即Cache-Control、Content-Language、Content-Type、Expires、Last-Modified、Pragma。

Expires：包含一个Http时间戳，在此日期之后，响应过期。

Last-Modified：响应首部字段，包含源头服务器认定的资源做出修改的日期及时间。

Pragme：通用首部，效果依赖于不同的实现。

预检请求

与简单请求不同，需预检的请求要去必须首先使用OPTIONS向服务器发送一个预检请求，已获知服务器是否允许该实际请求。
